# Tax Model Schemas – Overview

This document explains the relationship between the tax data models and how to use them.

## Schema Hierarchy

| Schema | Size | Purpose | Use Case |
|-------|------|---------|----------|
| **biztax-savings/tax-model-schema.json** | ~3.8M chars | Full enterprise schema with LACERTE, TurboTax, Pro Series product mappings, nativeAddress, transformations | Tax engine integration, multi-product support |
| **tax-model-input-schema-annotated.json** | ~12K lines, 717 fields | Same structure, annotated with `human_readable` and `description`; strips TurboTax mappings | LLM-friendly reference, documentation |
| **tax-model-simplified.json** | 677 fields | Annotated schema with products/nativeAddress stripped; retains hierarchy, human_readable, description, valueType | Field discovery, mapping, bulk export |
| **tax-model-core.json** | ~80 fields | Minimal insight-ready subset; aligned with tax-situation-and-1040-schema | LLM extraction, calculators, insights |
| **tax-situation-and-1040-schema.json** | ~160 lines | JSON Schema for input (tax_situation) + output (form_1040_calculated_lines) | LLM extraction prompt, UI display contract |

## What Was Simplified

**Removed (noise):**
- `products` arrays (LACERTE/PTO/TurboTax/Pro Series)
- `nativeAddress` (INPUT_CODE, series, code, path, onMisc, override)
- `transformations` (mapSelected, etc.)
- Internal keys: `defaultSection`, `section1`, `wgTpSp`, strategy overrides
- Redundant `*absorbed` limit-tracking fields

**Kept (signal):**
- `human_readable` labels
- `description` for each field
- `valueType` (integer, string, boolean, real)
- Logical hierarchy: summary, income, adjustments, deductions, credits, payments, tax

## Recommended Usage

1. **For LLM extraction from raw text**  
   Use `tax-situation-and-1040-schema.json` or `tax-model-core.json`. Both have human-readable keys and descriptions; core adds more granular income/deduction structure.

2. **For insights and strategy**  
   Use `tax-model-core.json` + `tax_situation` input. The core schema covers the inputs and outputs needed for QBI phaseout, NIIT, stacking, credits.

3. **For mapping to a tax engine**  
   Use `tax-model-simplified.json` for field discovery; then reference `tax-model-input-schema-annotated.json` for `products`/`nativeAddress` when you need LACERTE/PTO addresses.

4. **For TurboTax/Pro Series mapping**  
   Use `biztax-savings/.../tax-model-schema.json`; it includes `transformations` for those products.
